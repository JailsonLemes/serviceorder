Chat Service — Implementação Completa

Sistema de chat em tempo real vinculado a ordens de serviço. Socket.io + Redis Pub/Sub + PostgreSQL. Chat liberado exclusivamente após transição para ACCEPTED.

1. Decisão Arquitetural

Socket.io com Redis Adapter é a escolha correta para este sistema. Firebase Realtime Database oferece infraestrutura gerenciada, mas cria vendor lock-in, dificulta integração com a FSM de ordens (que vive no PostgreSQL), e torna o controle de acesso baseado em estado muito mais complexo. WebSocket nativo exige construir do zero reconexão, rooms, fallback e acknowledgements — esforço alto sem ganho real.

Socket.io oferece: rooms nativas (uma por ordem), fallback automático para long-polling quando WebSocket não está disponível (redes corporativas, proxies), acknowledgements para entrega garantida, reconexão exponential backoff built-in, e o Redis Adapter (@socket.io/redis-adapter) que resolve escalabilidade horizontal sem sticky sessions.

2. Controle de Acesso — Chat só após ACCEPTED

O canal não existe até a transição para ACCEPTED. Quando o freelancer aceita, a tabela chat_channels é criada dentro da mesma transação de banco que registra a transição de status. Qualquer tentativa de conectar antes disso retorna erro.

⚠️ O ws_token tem validade de 5 minutos e é obtido via REST (GET /orders/:id/chat). O servidor Socket.io valida este token no middleware de conexão — se inválido ou expirado, a conexão é recusada imediatamente.

src/chat/middlewares/auth.middleware.ts
import { Socket } from 'socket.io';
import { verifyJwt } from '../auth/jwt';
import { db } from '../database';

export async function chatAuthMiddleware(
  socket: Socket,
  next: (err?: Error) => void
) {
  const token = socket.handshake.auth.token;

  if (!token) return next(new Error('WS_TOKEN_REQUIRED'));

  let payload;
  try {
    // ws_token tem tipo 'chat' no payload para distinguir do access_token
    payload = verifyJwt(token);
    if (payload.type !== 'chat') throw new Error();
  } catch {
    return next(new Error('WS_TOKEN_INVALID'));
  }

  // Verifica se o canal existe e está ativo (ordem em ACCEPTED ou posterior)
  const channel = await db.query(`
    SELECT c.id, c.order_id, c.status, c.company_id, c.freelancer_id,
           o.status AS order_status
    FROM chat_channels c
    JOIN orders o ON o.id = c.order_id
    WHERE c.id = $1
      AND c.status = 'ACTIVE'
      AND o.status NOT IN ('DRAFT','PUBLISHED','IN_APPLICATION','SELECTED')
  `, [payload.channel_id]);

  if (!channel.rows.length)
    return next(new Error('CHAT_NOT_AVAILABLE'));

  const ch = channel.rows[0];

  // Verifica se o usuário é participante desta ordem
  const isParticipant =
    (payload.role === 'EMPRESA'     && ch.company_id    === payload.profile_id) ||
    (payload.role === 'FREELANCER'  && ch.freelancer_id === payload.profile_id);

  if (!isParticipant)
    return next(new Error('NOT_CHAT_PARTICIPANT'));

  // Anexa dados ao socket para uso nos event handlers
  socket.data = {
    userId:    payload.user_id,
    role:      payload.role,
    profileId: payload.profile_id,
    channelId: ch.id,
    orderId:   ch.order_id,
  };

  next();
}

3. Protocolo de Eventos WebSocket

Todos os eventos seguem o padrão domínio:ação. Eventos C→S são enviados pelo cliente; S→C são emitidos pelo servidor; SYS são gerados automaticamente pelo sistema.

EventoDir.PayloadDescrição
chat:message:sendC→S{content, content_type, reply_to_id?, attachment_keys[], client_msg_id}Enviar mensagem. client_msg_id garante idempotência em retries.
chat:message:newS→C{id, content, sender, content_type, attachments[], created_at, client_msg_id}Nova mensagem (emitida para todos na sala, incluindo remetente).
chat:message:ackS→C{client_msg_id, server_msg_id, created_at}Confirmação de recebimento para o remetente. Mapeia ID do cliente para ID do servidor.
chat:message:readC→S{message_id}Marca como lida. Marcar a mais recente marca todas as anteriores.
chat:message:readS→C{message_id, reader_id, read_at}Notifica o outro participante sobre a leitura.
chat:message:editC→S{message_id, content}Editar. Janela de 15 minutos após criação.
chat:message:editedS→C{message_id, content, edited_at}Broadcast para a sala após edição bem-sucedida.
chat:message:deleteC→S{message_id}Soft delete. Content zerado; registro permanece.
chat:message:deletedS→C{message_id, deleted_at}Broadcast para a sala após exclusão.
chat:typing:startC→S{}Throttle client-side: enviar no máximo 1x a cada 3s.
chat:typing:stopC→S{}Enviar ao parar de digitar ou ao enviar mensagem.
chat:typingS→C{user_id, role, is_typing}Broadcast para outros na sala. Auto-timeout de 5s via Redis TTL.
chat:channel:closedSYS{reason, order_status}Canal fechado (ordem COMPLETED, CANCELLED ou DISPUTED). Clientes devem desconectar.
chat:errorS→C{code, message}Erros específicos do chat (rate limit, permissão, etc.).
chat:connectedS→C{channel_id, unread_count, last_message_at}Emitido ao conectar com sucesso. Cliente usa para sincronizar estado.

4. Implementação do Servidor

src/chat/chat.gateway.ts
import { Server, Socket } from 'socket.io';
import { createAdapter } from '@socket.io/redis-adapter';
import { createClient } from 'redis';
import { chatAuthMiddleware } from './middlewares/auth.middleware';
import { MessageService } from './services/message.service';
import { TypingService } from './services/typing.service';
import { rateLimiter } from './middlewares/rate-limit.middleware';
import { auditLog } from '../audit/audit.service';

export async function bootstrapChatServer(httpServer: any) {
  // Redis clients separados para pub e sub (requisito do redis-adapter)
  const pubClient = createClient({ url: process.env.REDIS_URL });
  const subClient = pubClient.duplicate();
  await Promise.all([pubClient.connect(), subClient.connect()]);

  const io = new Server(httpServer, {
    adapter: createAdapter(pubClient, subClient),
    cors: { origin: process.env.ALLOWED_ORIGINS.split(',') },
    // Ping/pong para detectar conexões mortas
    pingTimeout:  20000,
    pingInterval: 25000,
    // Limita tamanho do payload (previne mensagens gigantes)
    maxHttpBufferSize: 1e6,  // 1 MB
  });

  // Middleware global: autenticação e controle de acesso
  io.use(chatAuthMiddleware);

  io.on('connection', async (socket: Socket) => {
    const { userId, channelId, orderId, role } = socket.data;
    const room = `order:${orderId}`;

    // Entra na sala da ordem
    await socket.join(room);

    // Envia estado inicial ao cliente
    const unreadCount = await MessageService.getUnreadCount(channelId, userId);
    socket.emit('chat:connected', { channelId, unreadCount });

    // ── Enviar mensagem ──────────────────────────────────
    socket.on('chat:message:send', async (payload, ackFn) => {
      // Rate limit: 10 mensagens/min por usuário
      const limited = await rateLimiter(`msg:${userId}`, 10, 60);
      if (limited) {
        socket.emit('chat:error', { code: 'RATE_LIMITED' });
        return;
      }

      try {
        // Salva no PostgreSQL. client_msg_id evita duplicatas em retry.
        const message = await MessageService.create({
          channelId,
          senderId:     userId,
          senderRole:   role,
          content:      sanitize(payload.content),
          contentType:  payload.content_type,
          replyToId:    payload.reply_to_id,
          attachmentKeys: payload.attachment_keys ?? [],
          clientMsgId:  payload.client_msg_id,
        });

        // ACK imediato para o remetente (mapeia client_id → server_id)
        if (typeof ackFn === 'function')
          ackFn({ server_msg_id: message.id, created_at: message.created_at });

        // Broadcast para TODOS na sala (incluindo remetente via redis adapter)
        io.to(room).emit('chat:message:new', message);

        // Auditoria assíncrona
        auditLog({
          entityType: 'chat_message', entityId: message.id,
          action:     'message.created', actorId: userId,
          afterData: { channelId, orderId, contentType: payload.content_type },
        });
      } catch (err) {
        socket.emit('chat:error', { code: 'MESSAGE_SEND_FAILED' });
      }
    });

    // ── Leitura ──────────────────────────────────────────
    socket.on('chat:message:read', async ({ message_id }) => {
      const read_at = await MessageService.markRead(message_id, userId, channelId);
      // Notifica o outro participante
      socket.to(room).emit('chat:message:read', { message_id, reader_id: userId, read_at });
    });

    // ── Indicador de digitação ────────────────────────────
    socket.on('chat:typing:start', async () => {
      // SETEX typing:{channelId}:{userId} 5 "1" — auto-expira em 5s
      await TypingService.setTyping(channelId, userId);
      socket.to(room).emit('chat:typing', { user_id: userId, role, is_typing: true });
    });

    socket.on('chat:typing:stop', async () => {
      await TypingService.clearTyping(channelId, userId);
      socket.to(room).emit('chat:typing', { user_id: userId, role, is_typing: false });
    });

    // ── Desconexão ───────────────────────────────────────
    socket.on('disconnect', async (reason) => {
      await TypingService.clearTyping(channelId, userId);
      socket.to(room).emit('chat:typing', { user_id: userId, is_typing: false });
    });
  });

  return io;
}

5. Persistência e Entrega Garantida

A mensagem é sempre salva primeiro no PostgreSQL, depois emitida. Nunca o contrário. Isso garante que uma mensagem que chegou ao servidor sempre pode ser recuperada via REST mesmo que o WebSocket caia antes do broadcast.

src/chat/services/message.service.ts
export class MessageService {
  static async create(input: CreateMessageInput) {
    return db.transaction(async (trx) => {
      // ON CONFLICT (client_msg_id) DO NOTHING — idempotente em retries
      const result = await trx.query(`
        INSERT INTO chat_messages
          (channel_id, sender_id, sender_role, content, content_type,
           reply_to_id, client_msg_id)
        VALUES ($1,$2,$3,$4,$5,$6,$7)
        ON CONFLICT (client_msg_id) DO UPDATE
          SET client_msg_id = EXCLUDED.client_msg_id  -- no-op, retorna existing
        RETURNING *
      `, [
        input.channelId,    input.senderId,
        input.senderRole,  input.content,
        input.contentType, input.replyToId,
        input.clientMsgId,
      ]);

      const message = result.rows[0];

      // Associa arquivos validando que pertencem a este canal
      if (input.attachmentKeys.length) {
        await AttachmentService.linkToMessage(
          trx, message.id, input.attachmentKeys, input.channelId
        );
      }

      // Atualiza contadores do canal (trigger já faz isso, mas retornamos o atualizado)
      await trx.query(`
        UPDATE chat_channels
        SET last_message_at = $1, message_count = message_count + 1
        WHERE id = $2
      `, [message.created_at, input.channelId]);

      return message;
    });
  }

  static async markRead(messageId: string, userId: string, channelId: string) {
    // Marca a mensagem indicada E todas as anteriores não lidas
    const result = await db.query(`
      INSERT INTO chat_message_reads (message_id, user_id)
      SELECT m.id, $2
      FROM chat_messages m
      WHERE m.channel_id = $3
        AND m.created_at <= (
          SELECT created_at FROM chat_messages WHERE id = $1
        )
        AND m.sender_id != $2              -- não marca as próprias mensagens
        AND NOT EXISTS (
          SELECT 1 FROM chat_message_reads r
          WHERE r.message_id = m.id AND r.user_id = $2
        )
      ON CONFLICT DO NOTHING
      RETURNING NOW() AS read_at
    `, [messageId, userId, channelId]);

    return result.rows[0]?.read_at ?? new Date().toISOString();
  }
}

6. Estratégia de Reconexão no Cliente

O cliente precisa lidar com três cenários: conexão inicial, reconexão após queda, e reabertura do app. Em todos os casos, a estratégia é a mesma: buscar mensagens não recebidas via REST usando after_id como cursor.

src/chat/chat.client.ts (React Native / React)
class ChatClient {
  private socket: Socket | null = null;
  private lastReceivedId: string | null = null;
  // Fila local de mensagens pendentes (enviadas mas sem ACK)
  private pendingMessages: Map<string, PendingMessage> = new Map();

  async connect(wsToken: string, wsUrl: string) {
    this.socket = io(wsUrl, {
      auth: { token: wsToken },
      // Reconexão automática com backoff exponencial
      reconnectionDelay:    1000,
      reconnectionDelayMax: 30000,
      reconnectionAttempts: 15,
      transports: ['websocket', 'polling'],
    });

    this.socket.on('chat:message:new', (msg) => {
      this.lastReceivedId = msg.id;
      this.handleNewMessage(msg);
    });

    // Ao reconectar: busca mensagens perdidas durante a desconexão
    this.socket.on('reconnect', async () => {
      if (this.lastReceivedId) {
        // REST API: GET /orders/:id/chat/messages?after_id={last}
        const missed = await fetchMissedMessages(this.lastReceivedId);
        missed.forEach(msg => this.handleNewMessage(msg));
      }
      // Reenviar mensagens pendentes sem ACK
      this.pendingMessages.forEach(pending => this.sendWithAck(pending));
    });
  }

  sendWithAck(payload: MessagePayload) {
    const clientMsgId = generateUUID();
    // Adiciona à fila local com status 'sending'
    this.pendingMessages.set(clientMsgId, { ...payload, clientMsgId, status: 'sending' });

    // Socket.io acknowledgement: callback chamado pelo servidor após persistência
    this.socket?..emit(
      'chat:message:send',
      { ...payload, client_msg_id: clientMsgId },
      (ack: { server_msg_id: string }) => {
        // Remove da fila de pendentes ao receber ACK do servidor
        this.pendingMessages.delete(clientMsgId);
      }
    );
  }
}

7. Upload de Arquivos — Fluxo Completo

O servidor nunca é intermediário do upload. O cliente envia diretamente ao S3 via presigned URL. Isso elimina uso de banda e CPU do servidor para arquivos.

Solicitar URL: cliente faz POST /orders/:id/chat/attachments/presigned-url com nome, tamanho e MIME type. Servidor valida tipo permitido e tamanho (máx 100MB), gera URL presignada S3 com expiração de 5 min.
Upload direto: cliente faz PUT {upload_url} com o arquivo como body. Nenhuma autenticação adicional — a URL já contém as credenciais assinadas. Header Content-Type deve bater com o registrado.
Enviar mensagem: após upload confirmado (status 200 do S3), cliente envia evento chat:message:send via WebSocket com attachment_keys: [storage_key]. O servidor valida que o key existe no S3 antes de persistir.
Geração de thumbnail: para imagens, um job assíncrono (worker) processa o arquivo após salvar, gera thumbnail e atualiza chat_attachments.thumbnail_key. O evento chat:message:new já inclui thumbnail_url quando disponível.
Exibição: os arquivos são servidos sempre por CDN (CloudFront), nunca pelo S3 diretamente. URLs na resposta da API têm expiração de 1h para arquivos privados (presigned GET URLs).
src/chat/services/attachment.service.ts
const ALLOWED_MIME_TYPES = [
  'image/jpeg', 'image/png', 'image/webp', 'image/gif',
  'video/mp4',
  'application/pdf',
  'application/zip',
  'text/plain',
  'audio/mpeg', 'audio/ogg',
];

export class AttachmentService {
  static async generatePresignedUploadUrl({ orderId, channelId, userId, fileName, fileSizeBytes, mimeType }) {
    if (!ALLOWED_MIME_TYPES.includes(mimeType))
      throw new ValidationError('UNSUPPORTED_MIME_TYPE');

    if (fileSizeBytes > 100 * 1024 * 1024)
      throw new ValidationError('FILE_TOO_LARGE');

    // Key com UUID para evitar colisões e path traversal
    const key = `chat/${orderId}/${generateUUID()}/${sanitizeFileName(fileName)}`;

    const uploadUrl = await getSignedPutUrl({
      bucket:   process.env.S3_BUCKET,
      key,
      mimeType,
      expiresIn:  300,
      // Content-Length obrigatório: previne upload de arquivo diferente do declarado
      conditions: [['content-length-range', 1, fileSizeBytes]],
    });

    return { uploadUrl, storageKey: key };
  }

  static async linkToMessage(trx, messageId, keys: string[], channelId) {
    // Valida que os keys existem no S3 e pertencem a este canal
    for (const key of keys) {
      const orderId = key.split('/')[1];
      // Verifica que o order_id do key bate com o canal (previne uso de keys de outras ordens)
      const channelOrder = await trx.query(
        'SELECT order_id FROM chat_channels WHERE id = $1', [channelId]
      );
      if (channelOrder.rows[0].order_id !== orderId)
        throw new ValidationError('INVALID_FILE_KEY');
      // Verifica existência no S3
      await s3HeadObject(key);
    }

    await trx.query(`
      INSERT INTO chat_attachments (message_id, storage_key, file_name, mime_type, file_size_bytes)
      SELECT $1, key, split_part(key,'/',-1), mime_from_key(key), s3_size(key)
      FROM UNNEST($2::text[]) AS key
    `, [messageId, keys]);
  }
}

8. Notificações em Tempo Real — Fallback para Push

Quando o destinatário está offline (sem socket ativo), o evento WebSocket chega ao Redis Adapter mas não encontra nenhum socket na sala para aquele usuário. O servidor precisa detectar isso e disparar push notification como fallback.

src/chat/services/notification-fallback.service.ts
export async function notifyIfOffline(
  io: Server,
  room: string,
  recipientId: string,
  message: ChatMessage
) {
  // Verifica sockets ativos na sala para o destinatário
  const sockets = await io.in(room).fetchSockets();
  const isOnline = sockets.some(s => s.data.userId === recipientId);

  if (!isOnline) {
    // Publica na fila de notificações para o worker processar
    await queue.publish('notifications', {
      type:        'CHAT_MESSAGE',
      userId:      recipientId,
      title:       `Nova mensagem em ${message.orderCode}`,
      body:        truncate(message.content, 80),  // nunca expõe conteúdo completo no push
      data: {
        screen:    'OrderChat',
        orderId:   message.orderId,
        channelId: message.channelId,
      },
    });
  }
}

9. Escalabilidade — Redis Adapter e Múltiplas Instâncias

Sem Redis Adapter, cada instância Socket.io é uma ilha: se a empresa está conectada na instância A e o freelancer na instância B, a mensagem do freelancer nunca chega à empresa. Com o adapter, todos os eventos são roteados via Redis Pub/Sub automaticamente, sem sticky sessions.

Com Redis Adapter

Instância B recebe mensagem → persiste no PostgreSQL → publica no canal Redis socket.io#order:456 → Instância A lê do Redis → emite para socket conectado. Transparente para o cliente.

Gargalos e limites

Cada instância Node.js aguenta ~10.000 conexões simultâneas (ajuste ulimit). Para 100k usuários simultâneos: 10 pods. Redis suporta pub/sub com milhões de mensagens/s no modo Cluster.

10. Segurança e Auditoria

🔴 Nunca confiar no conteúdo das mensagens. Todo content passa por sanitização antes de persistir (remoção de HTML, links maliciosos, e conteúdo acima de 10k chars). Nunca executar código do cliente.

Controles de segurança implementados

  • ws_token de curta duração (5 min): mesmo que interceptado, a janela de uso é mínima. Renovação transparente via GET /orders/:id/chat.
  • Rate limiting por evento: 10 mensagens/min, 30 typing events/min, 5 edições/hora por usuário. Implementado via Redis INCR + EXPIRE.
  • Validação de participante em cada evento: o socket.data setado no middleware é autoritativo — não confiar em dados do payload do cliente.
  • Content-Length obrigatório no presigned URL: previne substituição do arquivo após geração da URL assinada.
  • maxHttpBufferSize: 1MB no Socket.io: previne mensagens maliciosas gigantes que esgotem memória.
  • Sanitização de nome de arquivo: remove path traversal (../), caracteres especiais, e normaliza o nome antes de compor o S3 key.

Auditoria imutável

Todo evento crítico grava na tabela audit_logs. Para o chat, os eventos auditados são:

AçãoDados registrados
chat.channel_createdorder_id, triggered_by (ordem aceita), timestamp
chat.message_createdmessage_id, channel_id, order_id, content_type (nunca o conteúdo em si)
chat.message_deletedmessage_id, actor_id, razão do delete
chat.channel_closedorder_id, reason (order_status que causou o fechamento)
chat.unauthorized_accessuser_id tentante, order_id, motivo da rejeição

ℹ️ O conteúdo das mensagens nunca é gravado no audit_log — apenas metadados. O conteúdo completo está em chat_messages com soft-delete. Mensagens deletadas têm content = null mas o registro permanece para fins de moderação e disputas.

11. Fechamento do Canal

Quando a ordem transita para COMPLETED, CANCELLED ou DISPUTED, o canal deve ser fechado imediatamente. Isso é feito via evento publicado no sistema de mensageria interno:

src/chat/consumers/order-events.consumer.ts
// Worker que ouve eventos da FSM de ordens
queue.subscribe('order.status_changed', async (event) => {
  const CLOSE_ON_STATUS = ['COMPLETED', 'CANCELLED', 'DISPUTED'];

  if (!CLOSE_ON_STATUS.includes(event.to_status)) return;

  // Marca canal como CLOSED no banco
  await db.query(`
    UPDATE chat_channels
    SET status = 'CLOSED', closed_at = NOW()
    WHERE order_id = $1 AND status = 'ACTIVE'
  `, [event.order_id]);

  // Notifica todos os sockets conectados nesta sala
  io.to(`order:${event.order_id}`).emit('chat:channel:closed', {
    reason:       event.to_status,
    order_status: event.to_status,
  });

  // Faz io.to(room).disconnectSockets() após 3s (grace period)
  setTimeout(() =>
    io.in(`order:${event.order_id}`).disconnectSockets(true),
  3000);
});
