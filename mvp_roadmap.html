MVP — Definição Enxuta

O sistema completo projetado tem profundidade suficiente para escalar a centenas de milhares de usuários. O MVP precisa validar o loop de valor central — empresa contrata, freelancer executa, pagamento acontece — com o mínimo de complexidade técnica que ainda entregue confiança real ao usuário.

1. O que Entra no MVP

✅ Funcionalidades Essenciais

  • Auth: email + senha, JWT, refresh token. Sem OAuth no lançamento.
  • Cadastro: empresa (CNPJ + dados) e freelancer (CPF + skills). Verificação por email.
  • Ordens: criar, publicar, listar com busca por texto simples (PostgreSQL tsvector). Filtros básicos: categoria, valor, status.
  • Candidaturas: aplicar com mensagem, empresa seleciona, freelancer aceita ou rejeita.
  • FSM simplificado: 8 estados (sem ADJUSTMENT_REQUESTED, sem DISPUTED).
  • Chat por ordem: Socket.io com Redis adapter desde o início (custo zero, evita reescrita).
  • Evidências: freelancer envia fotos (S3) + descrição. Empresa aprova.
  • Pagamento: escrow com Stripe cartão. Um único gateway. PIX fica para depois.
  • Liberação automática: job cron 3 dias após aprovação.
  • Avaliação básica: overall_score (1-5) + comentário. Sem critérios detalhados.
  • Email transacional: aprovação, seleção, pagamento recebido. Sem push.
  • Admin mínimo: listar usuários, ver ordem, suspender conta, liberar pagamento manual.

🔴 Fora do MVP

  • PIX: adiciona gateway, webhook async e mTLS. Cartão resolve o problema.
  • Disputas: admin resolve manualmente por email/WhatsApp nas primeiras semanas.
  • Reputação Bayesiana: média simples é suficiente para validação.
  • Ranking avançado: ordenação por nota + número de avaliações resolve.
  • Elasticsearch: PostgreSQL full-text aguenta bem até 100k ordens.
  • App mobile: web responsivo primeiro.
  • Push notifications: email cobre tudo no lançamento.
  • ADJUSTMENT_REQUESTED: empresa aprova ou não. Negociação via chat.
  • Stripe Connect: transferência para freelancers via PIX manual no início.
  • Badges e achievements.
  • Fraud detection avançado: regras básicas apenas (nova conta + muitas avaliações).
  • Relatórios financeiros complexos: planilha manual serve.
  • Categorias hierárquicas: lista plana de 10-15 categorias.

💡 Princípio central do MVP: toda funcionalidade que não está no loop de valor (publicar → contratar → executar → pagar) fica fora. Disputas? Admin resolve no Slack. PIX? Empresa paga no cartão. Reputação sofisticada? Estrelinhas simples. O objetivo é provar que as partes se encontram e transacionam, não que o sistema é perfeito.

2. FSM Simplificado para o MVP

Redução de 12 para 8 estados. O que foi cortado: IN_APPLICATION (unificado em OPEN), SELECTED (unificado em IN_SELECTION), ADJUSTMENT_REQUESTED (não existe no MVP), DISPUTED (não existe). Isso elimina ~30% da complexidade da FSM e dos guards de transição.

4. Prioridade de Desenvolvimento — Sprints

Time de 2-3 desenvolvedores. Sprints de 2 semanas. MVP em 12 semanas (3 meses).

SPRINT 1-2
Fundação — Auth, usuários, banco de dados
Semanas 1–4
Entregar: cadastro de empresa e freelancer, login/logout, JWT + refresh, verificação de email, estrutura completa do banco de dados (DDL), CI/CD básico (GitHub Actions → deploy automático), ambiente de staging.

Decisão crítica: implementar o banco de dados já com o modelo completo (UUIDs, enums, audit_logs, soft delete) — essa base não muda. Mudar o schema depois custa caro.
SPRINT 3-4
Ordens e candidaturas
Semanas 5–8
Entregar: CRUD de ordens, publicação, listagem com busca (PostgreSQL full-text), candidatura do freelancer, seleção pela empresa, aceite pelo freelancer, FSM de 8 estados com guards corretos, histórico de status.

Decisão crítica: implementar a FSM na camada de serviço desde o início (não deixar lógica de transição no controller). Adicionar estados depois fica trivial.
SPRINT 5-6
Chat, evidências e aprovação
Semanas 9–12
Entregar: canal de chat por ordem (Socket.io + Redis), envio de evidências (upload para S3 via presigned URL), tela de revisão para empresa, fluxo de aprovação (→ APPROVED), avaliação simples após conclusão, email transacional para eventos críticos (seleção, aprovação, pagamento).

Decisão crítica: presigned URL para upload direto ao S3 — nunca passar arquivos pelo servidor da aplicação.
SPRINT 7-8
Pagamento e admin mínimo
Semanas 13–16
Entregar: integração Stripe (autorização na seleção, captura no aceite, job de liberação 3 dias após aprovação), webhook do Stripe com HMAC, painel admin básico (listagem de usuários + ordens, suspensão de conta, liberação manual de pagamento), transferência manual para freelancers via PIX pelo admin.

Decisão crítica: implementar o webhook com body raw e processamento na fila desde o início. Jamais processar webhook de forma síncrona.
SPRINT 9
Polimento, testes e lançamento
Semanas 17–18
Entregar: testes end-to-end do loop completo, pentest básico (OWASP Top 10), stress test do chat com k6, monitoramento com Sentry (erros) + UptimeRobot (uptime), rate limiting nas rotas críticas, políticas de privacidade + termos de uso, onboarding dos primeiros usuários beta.

Meta: 10 ordens concluídas com pagamento real no ambiente de produção antes do lançamento público.

5. Riscos Técnicos

RiscoSeveridadeProbabilidadeMitigação
Compliance de pagamento (PCI DSS)
Dados de cartão chegando ao servidor		 CRÍTICO Média Usar Stripe.js / Elements no frontend — número do cartão nunca toca o backend. Certificação PCI SAQ A (mais leve). Nunca implementar campo de cartão customizado.
LGPD
CPF, dados bancários, fotos de evidências		 CRÍTICO Alta DPO definido antes do lançamento. Política de privacidade revisada por advogado. Dados sensíveis (CPF, chave PIX) criptografados em repouso. Mecanismo de exclusão de conta que apaga dados pessoais.
Fraude no pagamento
Chargebacks sem sistema de fraude		 ALTO Baixa (volume pequeno) Limite de valor por ordem no MVP (ex: máx R$ 5.000). Liberar escrow somente após aprovação explícita da empresa. Monitorar chargebacks no Stripe dashboard manualmente.
FSM crescendo sem controle
Lógica de transição espalhada		 ALTO Alta (sem disciplina) FSM implementada em classe centralizada (OrderStateMachine). Tabela de transições permitidas como constante. Nenhuma transição fora desta classe. Guards validados sempre na camada de serviço.
Chat perdendo mensagens
Reconexão sem backfill		 ALTO Média Persistir mensagem no PostgreSQL antes de fazer broadcast. Cliente envia after_id na reconexão para buscar mensagens perdidas. Testar desconexão e reconexão nos testes E2E.
Transferências manuais para freelancers
Admin esquece de pagar		 ALTO Alta (processo manual) Job diário lista pagamentos em status RELEASED sem transferência confirmada e envia email de alerta para admin. Planilha de controle até implementar Stripe Connect / PIX automático.
Sem sistema de disputas
Conflito sem protocolo		 MÉDIO Certa (vai acontecer) SLA de resposta por email de 24h. Admin tem ação manual de refund e release no painel. Limitar valor máximo por ordem no MVP reduz exposição. Priorizar sistema de disputas no V1.1.
Escalonamento do banco
Queries pesadas sem índices		 MÉDIO Baixa (volume MVP) DDL completo já inclui todos os índices críticos. PostgreSQL gerenciado com backups automáticos. EXPLAIN ANALYZE em toda query nova antes de ir para produção.

6. Roadmap Pós-MVP

7. O que NÃO Pode Ser Simplificado nem no MVP

🔴 Segurança de pagamento é inegociável. Stripe.js obrigatório. HMAC no webhook. Idempotency key em toda operação financeira. Sem atalho aqui — um double-charge ou dado de cartão exposto destrói a confiança antes do produto existir.

Banco de dados

UUIDs, TIMESTAMPTZ, NUMERIC(12,2) para dinheiro, soft delete, enums PostgreSQL. Mudar o tipo de dado de uma coluna em produção com dados reais é caro e arriscado.

FSM na camada de serviço

Toda transição de estado validada em OrderStateMachine. Nunca no controller, nunca no banco. Adicionar estado no futuro tem que ser trivial.

Histórico de status imutável

order_status_history com REVOKE UPDATE/DELETE. Em disputas e litígios, essa tabela é a única fonte de verdade auditável do que aconteceu.

Idempotência no pagamento

Idempotency key UUID obrigatória em toda operação financeira. Retries sem idempotência resultam em double-charge — impossível de explicar para o cliente.

Chat persistente antes do broadcast

INSERT no banco antes de emitir para o socket. Usuário perde conexão e a mensagem tem que estar lá quando reconectar. Se a ordem for inversa, mensagens desaparecem.

Rate limiting desde o início

Login (5/min), candidatura (10/hora), pagamento (5/hora por empresa). Sem isso, o sistema vira alvo fácil de bots no primeiro dia de divulgação.

8. Critérios de Saída do MVP

O MVP está pronto para lançamento público quando todas as condições abaixo forem verdadeiras:

checklist de lançamento
Loop completo funcional:
☑ Empresa cria conta → publica ordem → recebe candidaturas → seleciona
☑ Freelancer candidata → aceita → chat abre → envia evidências
☑ Empresa aprova → pagamento liberado 3 dias após → avaliação enviada
☑ 10 ordens reais concluídas em staging com cartão real (Stripe test mode)

Pagamento:
☑ Stripe.js no frontend (número do cartão nunca chega ao backend)
☑ Webhook com HMAC validado + processamento na fila
☑ Job de liberação testado com `FOR UPDATE SKIP LOCKED`
☑ Refund manual funciona no painel admin

Segurança básica:
☑ Rate limiting em login, candidatura e pagamento
☑ OWASP Top 10 revisado (SQL injection, XSS, IDOR nas ordens)
☑ Arquivos de upload validados por mime type (não por extensão)
☑ JWT com expiração curta (15min access, 7d refresh)

Operação:
☑ Sentry configurado (erros chegam no Slack em < 1 min)
☑ Uptime monitor com alertas (UptimeRobot ou similar)
☑ Backup automático do PostgreSQL verificado (restore testado)
☑ Runbook de incidents: o que fazer se Stripe cair, se DB cair, se chat cair
☑ Admin sabe como fazer refund manual e release manual
☑ Email transacional funcionando para todos os eventos críticos

Legal:
☑ Política de privacidade publicada (LGPD)
☑ Termos de uso publicados
☑ Mecanismo de exclusão de conta implementado

💡 Estratégia de lançamento: não lançar para o público geral imediatamente. Fazer um beta fechado com 5-10 empresas parceiras e 20-30 freelancers selecionados. Acompanhar cada ordem manualmente nas primeiras semanas. Isso permite corrigir bugs críticos antes que eles afetem centenas de usuários — e garante que as primeiras avaliações do produto sejam positivas.
